Micro Service 2 HLD

Overview

  • This micro-service is responsible for:
    • Generating a user summary based on answered questions.
    • Running chat between two persona based bots.
  • Unlike 1:1 chat between people where there is direct message passing between interested parties, the chat for the persona based chatbots used here would be indirect with Gemini API acting as the broker.
  • Each message goes through Gemini API along with a system prompt specifying what to do with the message. The response from the API will be fed again to the API with a different prompt (discussed later).

Endpoints

1. POST /intitiate/{fromUid}/{toUid}

Overview

  • This endpoint is responsible for initiating chatbot chat between two users.
  • This will trigger a workflow that will start a chat between two user's persona based chatbots.
  • This entire operation can be thought of as a Finite State Machine (details below).

State Machine Description

States
  1. CO: Commence, this will be the initial state on hitting the /initiate endpoint.
  2. COMM: Communicate, this state will be similar to CO but is for any other communication between agents after the first one.
  3. ASK_G: Ask Gemini, this will use Gemini API to get answers that might be in a chat message, given the target user's summary.
  4. ASK_U: Ask User, this will be when Gemini fails to extract an answer from the target user's summary. These questions will be fed to the actual user to be answered.
  5. TERM: Terminal, when a terminal condition is met. A terminal condition is either:
    a. When the interests extracted from summary match 75%.
    b. When the number of messages passed between two user's agents is equal to 50 (i.e. 25 messages each).
Transitions
  1. NOQ: NO Question. When on a state there is no questions to be asked to the other user's agent.
  2. HAQ: HAve Question. When a chat message contains questions to be answered. The questions can be answered directly from user's previously answered questions or may be an entirely new question that has to be asked to the user.
  3. HAA: HAve Answer. When the user has answered the questions presented to them.
  4. MCM: Match Condition Met. If either of the (aforementioned) two conditions met.
  5. MCN: Match Condition Not met. If the conditions (aforementioned) two conditions are not met.
Explanation
  • On chat initiation from user u1 to user u2, it goes into CO state. This marks the commencement of the chat state machine.
  • If on the initial conversation from u1 to u2, if there are questions for u2, then it transitions into ASK_Gwhich then passes the questions along with u2's summary for Gemini to answer the questions if possible.
  • If some questions cannot be answered from summary, it transitions to ASK_U state. This asks the questions directly to user u2.
  • The next state after both ASK_G and ASK_U after getting the answer from either Gemini or the actual User is COMM which is basically CO but with previous user's message apart from both user's summaries.
  • After executing COMM the conditions are checked, if they are not met then it transitions back to COMM otherwise moves to the terminal state TERM.

Design

  • The API itself is stateless, and the state will be passed while transitioning between steps. This decreases the memory usage by not having to maintain states across thousands of active chats at one time.
  • The state machine will have a RabbitMQ queue in front of it, along with multiple brokers responsible for accepting the state and executing the transition function.
  • The queue ensures proper load balancing as chat states will be a high volume operation.
